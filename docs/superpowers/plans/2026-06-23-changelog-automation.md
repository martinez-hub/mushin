# Changelog Automation (towncrier) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Adopt towncrier news fragments so every PR documents its own change, enforce it with a CI gate, and assemble fragments into `CHANGELOG.md` at release time.

**Architecture:** Add `[tool.towncrier]` config + a `changes/` fragment directory; a new `changelog` CI job runs `towncrier check` on PRs; a `make changelog` target runs `towncrier build` at release. No application code changes. Spec: `docs/superpowers/specs/2026-06-23-changelog-automation-design.md`.

**Tech Stack:** towncrier (Markdown output), GitHub Actions, uv, Make.

**Context for the implementer:** This is the `mushin` repo at `~/code/mushin` (distribution name `mushin-py`, import `mushin`). Branch is `changelog-towncrier`, already cut from `main`, and already carries an uncommitted `uv.lock` change (the self-version `0.1.0`→`0.2.0` sync — keep it; it's intentional and gets a seed fragment in Task 6). The existing `CHANGELOG.md` follows Keep a Changelog with `## [Unreleased]`, `## [0.2.0] - 2026-06-23`, `## [0.1.0] - 2026-06-22`, and a link-reference footer. CI lives in `.github/workflows/ci.yml` with `lint` and `test` jobs. **Never add Claude/AI authorship to any commit, file, or doc.** The towncrier config below was verified against a copy of our real CHANGELOG during design — use it verbatim.

- [ ] **Step 2: Write the contributor README**

Create `changes/README.md`:

```markdown
# News fragments

Every pull request that changes user-facing behavior must add a **news
fragment** here. At release time these are assembled into `CHANGELOG.md` by
[towncrier](https://towncrier.readthedocs.io/) (`make changelog VERSION=X.Y.Z`).

## How to add one

Create a file named `<id>.<type>.md`, where `<id>` is your PR or issue number
(or `+<slug>` if you don't have one yet), and `<type>` is one of:

| type | CHANGELOG section | use for |
| --- | --- | --- |
| `added` | Added | new features |
| `changed` | Changed | changes to existing behavior |
| `fixed` | Fixed | bug fixes |
| `removed` | Removed | removed features |
| `deprecated` | Deprecated | soon-to-be-removed features |
| `misc` | Misc | CI/refactor/typo PRs with nothing user-facing |

The file holds one line of Markdown describing the change, written for users:

```
changes/123.fixed.md
```
```
`compare` no longer crashes on single-seed inputs.
```

A `misc` fragment can be empty — it only records that a PR happened (e.g.
`changes/+ci.misc.md`), satisfying the CI gate without adding a changelog line.

## Preview

`make changelog-draft VERSION=0.3.0` renders what the next release section will
look like, without consuming the fragments.
```

### Task 5: CI changelog-gate job

**Files:**
- Modify: `.github/workflows/ci.yml` (add a `changelog` job)

- [ ] **Step 1: Add the job**

Add this job to `.github/workflows/ci.yml` under `jobs:` (after the `test` job, same indentation as `lint`/`test`):

```yaml
  changelog:
    runs-on: ubuntu-latest
    # PRs only (the check diffs the base branch); never on push to main, and
    # exempt Dependabot (lockfile/deps PRs carry no user-facing fragment).
    if: github.event_name == 'pull_request' && github.actor != 'dependabot[bot]'
    steps:
      - uses: actions/checkout@v7
        with:
          fetch-depth: 0          # towncrier check diffs against the base branch
      - uses: astral-sh/setup-uv@v7
        with:
          enable-cache: true
      - name: Sync dev environment
        run: uv sync
      - name: Changelog fragment check
        run: uv run towncrier check --compare-with origin/${{ github.base_ref }}
```

- [ ] **Step 2: Verify the YAML is valid**

Run: `uv run python -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))" && echo OK`
Expected: `OK` (no parse error).

- [ ] **Step 3: Verify the check command passes locally on this branch**

This branch has seed fragments, so the real `towncrier check` the CI job runs should pass against `main`:

Run: `git fetch origin main 2>/dev/null; uv run towncrier check --compare-with origin/main`
Expected: a "Found:" line listing the seed fragments, exit 0. (If `origin/main` isn't fetchable in the sandbox, compare against the local base instead: `uv run towncrier check --compare-with main`.)

# Releasing mushin

Releases are published to PyPI by `.github/workflows/publish.yml` when a GitHub
Release is published whose tag matches the project version.

## Steps

1. Choose the next version `X.Y.Z` (SemVer; `0.x` keeps breaking changes in the
   minor slot).
2. Assemble the changelog from news fragments:
   ```bash
   make changelog VERSION=X.Y.Z
   ```
   This rewrites `CHANGELOG.md` (a new `## [X.Y.Z] - <today>` section under the
   towncrier marker) and deletes the consumed fragments. Review the diff;
   tidy wording if needed.
3. Bump the version in `pyproject.toml` to `X.Y.Z`, then `uv sync` to update
   `uv.lock`.
4. Update the `[Unreleased]`/version link-reference footer at the bottom of
   `CHANGELOG.md`.
5. Commit on a branch, open a PR, merge once CI is green.
6. Tag and publish a GitHub Release `vX.Y.Z` on the merge commit. `publish.yml`
   verifies `tag == version` and publishes via Trusted Publishing.

Preview the changelog section any time without consuming fragments:

```bash
make changelog-draft VERSION=X.Y.Z
```
```
